**Role**: Owner + Veterinarian  
**Needs**:

### 2. Receptionist
**Role**: Reservation Management & Customer Interface  
**Needs**:
- Zero-overbooking reservation system
- Check-in/check-out efficiency
- Animal history access for personalization
- Payment processing integration

### 3. Caretakers
**Role**: Animal Care & Daily Operations  
**Needs**:
- Morning shift: Complete care plan visibility
- Real-time: Care registration with timestamps
- Communication: Structured notes between shifts
- Alerts: Health changes for veterinarian

### 4. Cleaner
**Role**: Accommodation Maintenance  
**Needs**:
- List of accommodations needing cleaning
- Mark completion status
- Immediate visibility to reception

### 5. Veterinarian
**Role**: Clinical Health Management  
**Needs**:
- Complete clinical history per animal
- Intervention/prescription registration
- Alerts for health status changes
- Cost tracking for invoicing

## AI Instructions for LLM Assistance

### Context Injection (ALWAYS Include)
When asking for requirements/design/code:

1. **Mention Project**: "For LI4 hotel management system"
2. **Specify User**: "For non-technical staff who work shifts"
3. **Include Constraint**: "Real-time availability with < 2 sec response"
4. **Reference Standard**: "Following IEEE 830 specification"

### Output Format Requirements

#### For Requirements
- Use markdown with clear hierarchy
- Include acceptance criteria (testable, measurable)
- Reference stakeholder (which user needs this)
- Cite business problem solved
- No ambiguous language

#### For Architecture/Design
- Include rationale for decisions
- Document trade-offs considered
- Reference constraints addressed
- Suggest PoC (Proof of Concept) approach
- Link to requirements (RF-XX)

#### For Code
- Portuguese comments for business logic
- English comments for technical code
- Include error handling (what if payment fails?)
- Add edge cases (overlapping bookings, concurrent users)
- SOLID principle adherence
- Performance considerations

#### For Tests
- Happy path + error scenarios + edge cases
- Reference requirement (RF-XX) being verified
- Clear test names describing what's being tested
- Include performance assertions where applicable


### Language Preferences

- **Business Logic**:     Portuguese (comentários explicam regras de negócio)
- **Code Comments**:      Portuguese (detalhes técnicos e de implementação)

### Do's

## Special Considerations

### The Availability Control Problem (RF-05)
This is the MOST CRITICAL requirement:
- **Challenge**: Real-time checking under concurrent access
- **Complexity**: Must consider reservations + active stays + cleaning status
- **Performance**: < 100ms, 95th percentile < 2 seconds
- **Consistency**: No overbookings, ever
- **When Asking About This**: Mention the concurrency challenge

### Multi-Role Design
Five different user types with different information needs:
- **Receptionist**: Fast, intuitive for quick reservation
- **Caretaker**: Complete care details, structured communication
- **Cleaner**: Simple task list, status marking
- **Veterinarian**: Clinical history, alert system
- **Director**: Aggregate reporting, financial overview

**When Suggesting Features**: Consider which role benefits

### Future 24/7 Operation
Current system operates business hours only. Design should:
- Support future 24/7 without major restructuring
- Plan for night shift notifications
- Consider unattended check-in/check-out

**When Designing**: Mention this future scenario
